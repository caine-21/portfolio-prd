# 产品需求文档（PRD）
## AI 写作助手 — B 端 SaaS 平台内容生产模块

| 版本 | 状态 | 日期 | 作者 |
|---|---|---|---|
| v2.0 | 草稿 | 2025-04 | 产品经理 |

---

## 一、背景与问题定义

### 1.1 用户真实痛点

以电商运营团队为典型场景：

- 单篇商品详情页文案人工撰写耗时 30–60 分钟，SKU 数量增长时产能线性瓶颈
- 多人协作时风格漂移严重，同一品牌不同人写的文案语气不一致，品牌形象受损
- 使用 ChatGPT 等通用工具生成的内容频繁出现与企业产品规格不符的事实错误，需要大量核查时间

**核心矛盾：** 用户需要大量、稳定、符合品牌规范的内容，但人工生产慢、AI 生产不可信。

### 1.2 现有替代方案的不足

| 替代方案 | 用户现在怎么用 | 核心缺陷 |
|---|---|---|
| 纯人工撰写 | 运营逐篇写，leader 审 | 慢、风格不统一、无法规模化 |
| 通用 LLM（ChatGPT / 文心）| 粘贴需求进对话框 | 输出不含企业私有数据，事实错误率高 |
| 模板工具（秀米、135 编辑器）| 选模板填字 | 只解决格式，不解决内容生成 |

**现有方案共同失败点：** 没有把企业私有知识（产品库、品牌规范、历史优秀案例）注入生成环节。

### 1.3 机会假设

> 如果把企业私有知识通过 RAG 注入 LLM 生成，并在输出端加人工审核闭环，可以在保证内容可信度的前提下，将内容生产时间缩短 60%。

这是本产品的核心赌注。所有功能设计都服务于验证这一假设。

## 二、目标用户

### 2.1 用户分层

| 用户角色 | 典型场景 | 核心诉求 | 技术能力 |
|---|---|---|---|
| 内容运营 | 批量生成商品详情页文案 | 快、准、格式统一 | 低 |
| 市场经理 | 营销推文、报告摘要 | 品牌语气一致 | 中 |
| 知识管理员 | 维护企业知识库 | 检索准确，不出幻觉 | 中 |
| 管理员 | 配置模板、权限管理 | 可控、可审计 | 高 |

### 2.2 核心用户深度画像

**内容运营专员（主要服务对象）**

典型一天：早上开选品会，确定本周推广 SKU（约 20 个），下午全部要在系统里上线，每个 SKU 需要标题 + 卖点 + 详情三段文案。当前方式：翻历史文案参考，逐个改，改完发群里让 leader 过，来回两三轮，经常卡到晚上。

**JTBD：** 当我需要批量产出符合品牌规范的文案时，我希望系统帮我生成可用的初稿，让我只做最终确认——而不是从零开始写，也不是花大量时间核查 AI 是否说了不存在的功能参数。

**当前工作流痛点最深的地方：** 不是"生成慢"，而是"生成完还要花更多时间核查真实性"。这决定了产品的核心设计重心在**质量可信，而不仅仅是速度快**。

## 三、产品目标

### 3.1 北极星指标

**内容直接可用率 ≥ 75%**（用户发布前无内容修改操作，通过操作日志统计）

选这个指标的逻辑：直接可用率同时反映了生成质量（内容准确）和产品体验（用户信任输出）。当前人工内容直接发布率约 50–60%（行业估算基线），AI 辅助目标提升 15–25pp。

### 3.2 MVP 验证目标（上线后 30 天）

- 直接可用率 ≥ 75%
- 单篇内容生产时间 < 10 分钟（从打开到进入审核队列）
- 低置信度触发率 < 15%（验证知识库覆盖度足够）

### 3.3 非目标

- 自动发布（所有内容必须经过人工确认）
- C 端用户场景
- 完全自主 Agent（本期保留人工审核环节）
- 内容发布后的效果追踪（转化率、点击率）

## 四、核心用户旅程

这一章是本 PRD 的主轴。所有功能设计从这里出发。

### 4.1 主流程（Happy Path）

```
运营选择内容模板（如"商品详情页文案"）
  ↓
填写结构化表单（产品名、卖点、目标受众、字数）
  ↓
系统检索企业知识库（RAG，租户隔离）
  ├─ 命中 → 注入 chunks，标注来源
  └─ 未命中 → 继续生成，打「低置信度」标记
  ↓
LLM 生成 2 个候选版本（流式输出）
  ↓
质量评分自动计算（接地性 + 格式合规 + 风格一致性）
  ↓
进入审核队列（展示评分 + 来源标注）
  ↓
运营审核：选择版本 → 轻度修改（或直接通过）
  ↓
发布 / 归档（支持事后撤回，版本历史保留 90 天）
```

**关键时刻（Moment of Truth）：** 步骤"审核"——如果质量评分能让运营在 30 秒内判断"是否可用"，产品就成立；如果评分不可信，运营还是要逐字核查，产品等于没用。

### 4.2 失败路径与兜底

| 失败场景 | 触发条件 | 系统行为 |
|---|---|---|
| 知识库未命中 | 检索相关度 < 0.6 | 打「低置信度」标记，提示核查，不阻断 |
| 生成内容含事实错误 | 接地性评分 < 2 | 强制二次确认，不允许直接通过 |
| LLM 超时 | 响应 > 30s | 提示中断，支持重试，任务状态保留 |
| 敏感词命中 | 内容安全 API 扫描命中 | 自动打回，不进入发布队列 |
| 错误内容已发布 | 人工发现问题 | 一键撤回，版本历史可追溯 |

### 4.3 系统闭环（关键设计）

本产品的长期价值不只是"生成"，而是**随使用持续变好**：

```
用户输入（表单 + 知识库）
  → LLM 生成
  → 质量评分（接地性 / 格式合规 / 风格一致性）
  → 人工审核（通过 / 打回 + 问题标记）
       ↓
  反馈数据集（问题类型 + 修改内容）
       ↓
  ┌──────────────────────────────┐
  │  优化路径：                   │
  │  · 模板迭代（高打回率模板灰度调整）│
  │  · 知识库补充（低命中率告警）    │
  │  · 评分校准（标注数据 → Pearson）│
  └──────────────────────────────┘
```

**设计原则：** LLM 负责感知（生成 + 评分），代码负责决策（阈值判断 + 是否阻断），人工负责最终确认。三层分离，避免系统自己批准自己的输出。

## 五、功能需求

按用户旅程顺序组织。**MVP 范围：F1 + F2 + F3 + F4 + F5。F6 为第二期。**

### F1：内容模板管理（MVP）

管理员创建模板，定义字段、格式约束、示例内容。模板版本控制，支持回滚。新模板灰度发布（10% 用户先验证，稳定后全量）。

**为什么设计模板层：** 用户不具备 Prompt 工程能力，模板是把 PM 的产品判断封装成可复用资产的关键机制——用户填表单，PM / 管理员设计 Prompt，两者解耦。

### F2：结构化输入表单（MVP）

用户填写模板定义的字段（产品名、卖点、受众、字数限制）。必填字段校验。支持 CSV 批量导入触发批量生成任务。

### F3：RAG 知识库检索（MVP）

**功能：** 用户绑定企业知识库（产品手册、品牌规范、历史优秀案例）。生成时自动检索相关 chunk，注入 Prompt，输出内容标注来源，支持溯源查看。

**多租户隔离：** 每个客户独立向量索引，检索通过 workspace_id 严格限定范围，跨租户请求返回 403 并写入审计日志。

**设计决策 — 为什么不用全文搜索（BM25）而用向量检索：**
全文搜索依赖关键词匹配，在语义等价但措辞不同时（如"镜片折射率"vs"lens index"）命中率差。向量检索捕获语义相似度，更适合用户用自然语言提问的场景。代价是冷启动需要向量化计算，通过异步重建解决。

**边界：** 检索相关度 < 0.6 时打「低置信度」标记，预期触发率 < 15%。若连续 7 天触发率 > 30% 则系统告警，提示知识库覆盖度不足。

### F4：AI 生成与流式输出（MVP）

调用 LLM，流式返回降低等待感知，同时生成 2 个候选版本供用户选择。生成参数可配置：temperature、字数范围、语气风格。

**为什么生成 2 个版本而不是 1 个或 3 个：** 1 个版本用户没有对比感，容易直接接受低质量输出；3 个版本增加决策成本。2 个版本是最小对比基础，运营实际选择时更容易判断哪个更好。

### F5：人工审核与问题反馈（MVP）

生成结果进入待审核队列。编辑器支持 diff 高亮（AI 原文 vs 修改版）。审核操作：通过 / 打回重生成 / 标记问题类型（事实错误 / 格式问题 / 语气不符）。问题标记写入反馈数据集，驱动模板迭代。已发布内容支持一键撤回，版本历史保留 90 天。

### F6：质量评分（第二期）

**三项指标：**

| 维度 | 评估方式 | 设计说明 |
|---|---|---|
| 接地性评分（1–5）| LLM-as-judge，基于检索上下文评估答案是否准确反映来源 | 选 LLM-judge 而不是余弦相似度：余弦只看词汇接近，不理解逻辑关系；LLM-judge 综合问题 + 上下文 + 答案三者，与人工判断相关性更高 |
| 格式合规率 | 规则检测：字数、结构字段 | 确定性规则，无需模型，延迟低 |
| 风格一致性 | 与历史通过内容的余弦相似度 | 冷启动期（历史数据 < 50 篇）不展示此项 |

**第二期原因：** MVP 先验证"生成 + 审核"主流程跑通，收集真实用户数据后再校准评分模型，避免用假数据调参。

## 六、关键边界条件

| 场景 | 触发条件 | 系统行为 |
|---|---|---|
| 知识库未命中 | 相关度 < 0.6 | 打标记，不阻断 |
| 接地性评分极低 | < 2 分 | 强制二次确认 |
| LLM 超时 | > 30s | 提示中断，支持重试 |
| 批量任务过载 | > 100 条 | 拆分队列，异步执行，邮件通知 |
| 知识库索引更新 | 管理员上传新文档 | 异步重建，期间显示「索引更新中」 |
| 跨租户请求 | workspace_id 不匹配 | 403，写审计日志 |

| 功能 | 验收条件 | 优先级 |
|---|---|---|
| 模板管理 | 管理员创建模板后，运营用户可见且可使用；版本回滚后旧版本失效 | P0 |

| 生成延迟 | P95 < 15s（1000 并发压测） | P0 |
| 审核队列 | 审核操作 < 1s 响应；问题标记实时写入反馈数据集 | P0 |
| 撤回 | 已发布内容撤回后前台不可见，版本历史可查 | P0 |
| 低置信度 | 相关度 < 0.6 时「低置信度」标记必须出现，不漏标 | P1 |

## 八、风险与降级

| 风险 | 影响 | 降级策略 |
|---|---|---|
| LLM 服务不稳定 | 高 | 多供应商接入，模型可切换；关键模板结果缓存 |
| RAG 幻觉未被审核人发现 | 高 | 接地性评分 < 2 强制二次确认；发布后支持撤回 |
| 低置信度触发率失控（> 30%）| 中 | 知识库健康度看板 + 告警；触发 KB 补充工作流 |
| 模板质量差导致大量打回 | 中 | 灰度发布；打回率 > 40% 自动暂停模板并通知管理员 |
