# AI 写作助手 — PRD

> B 端 SaaS 平台内容生产模块。解决的核心问题：**企业私有知识无法注入 AI 生成，导致输出不可信、无法规模化使用。**

设计重心不是"生成更快"，而是"生成可信"——通过 RAG 知识注入 + LLM-judge 质量评分 + 人工审核闭环，让非技术用户能稳定使用 AI 内容生产。

---

## 系统设计主线

```
用户输入（结构化表单 + 模板）
  ↓
RAG 检索（企业知识库，租户隔离）
  ↓
LLM 生成（流式，2 个候选版本）
  ↓
质量评分（接地性 / 格式合规 / 风格一致性）
  ↓
人工审核确认（不自动发布）
  ↓
发布 / 打回 + 问题标记
       ↓
  反馈数据集 → 模板迭代 / 知识库补充 / 评分校准
```

**三层分离原则：** LLM 负责感知（生成 + 评分），代码负责决策（阈值判断），人工负责最终确认。避免系统自己批准自己的输出。

---

## 两个关键设计决策

**F3：为什么用向量检索而不是全文搜索**
全文搜索依赖关键词匹配，语义等价但措辞不同时命中率差。向量检索捕获语义相似度，适合用户用自然语言表述需求的场景。阈值 0.6 未命中时打「低置信度」标记，不阻断生成但强制人工核查。

**F6：为什么用 LLM-as-judge 而不是余弦相似度**
余弦相似度只看词汇接近，不理解逻辑关系（答案和上下文词汇重叠但逻辑相反时误判为高质量）。LLM-judge 综合问题 + 检索片段 + 答案三者关系，与人工判断相关性更高。代价是延迟，因此放在非关键路径（审核辅助），不阻断主流程。

---

## 北极星指标

**直接可用率 ≥ 75%**（用户发布前无内容修改操作）

当前人工内容直接发布率约 50–60%（基线估算），AI 辅助目标提升 15–25pp。衡量方式：操作日志，非人工标注，避免标注一致性问题。

## 技术声明的实验依据

PRD 中的 RAG 设计和质量评分方案，来自以下两个实验项目的真实结论：

- [langchain-rag-demo](https://github.com/caine-21/langchain-rag-demo) — RAG pipeline + 多智能体评估实验：chunk_size A/B 对照、LLM-judge Grounding Score 设计、Policy A→B 迭代实验（Day 6/7）
- [video-prompt-qa](https://github.com/caine-21/video-prompt-qa) — 多 LLM 对抗性评估：5 类失败模式分类，16 个边界 case，score gaming 发现

PRD 中的每一个技术判断都有实验数据支撑，不是纸面推断。
